# SECOND-BRAIN OPERATING RULES

## Objetivo
Manter o segundo cérebro útil, limpo e operacional.

Este sistema existe para ajudar Friday e Alex a:
- pensar melhor
- lembrar o que importa
- operar com continuidade
- transformar contexto em execução

Não existe para acumular arquivos sem função.

## Regra principal
Antes de criar um novo arquivo, perguntar:
- isso realmente precisa existir?
- isso poderia viver melhor em um arquivo já existente?
- isso ajuda a decidir, operar, lembrar, vender ou executar?

Se a resposta for não, não criar.

## O que cada camada guarda

### 1. SOUL / STYLE / IDENTITY / USER
Guardam identidade, tom, relação e forma de operar.

Não usar para tarefas do dia a dia.

### 2. MEMORY
Guarda contexto durável.

Usar para:
- fatos importantes
- preferências estáveis
- contexto de negócios
- aprendizados que realmente importam

Não usar para fila operacional do dia.

### 3. PLAYBOOK
Guarda lógica de atuação recorrente.

Usar quando o valor está em repetir bem um padrão.

### 4. PRIORITIES
Guarda foco estratégico atual.

Usar para prioridades em curso, não para microtarefas.

### 5. TASKS
Guarda fila viva de execução.

Usar para trabalho em andamento e próximos passos operacionais.

### 6. MAPA
Guarda a arquitetura do cérebro.

Usar para mostrar onde cada tipo de coisa deve viver.

### 7. cerebro/empresa
Guarda contexto de negócio, posicionamento, tese, produtos e direção empresarial.

### 8. cerebro/areas
Guarda operações por função.

Exemplos:
- vendas
- atendimento
- retenção
- dados de clientes
- rotina comercial

### 9. cerebro/agents
Guarda papéis de agentes, personas e funções especializadas.

### 10. cerebro/skills
Guarda lógica de capacidades e fluxos reutilizáveis do negócio.

### 11. .skills/
Guarda skills operacionais do ecossistema OpenClaw.

Só criar skill quando existir repetição clara, fluxo reutilizável ou necessidade de instrução especializada.

## Quando criar arquivo novo
Criar arquivo novo apenas quando pelo menos uma destas condições for verdadeira:
- o assunto vai voltar muitas vezes
- o conteúdo precisa viver separado para manter clareza
- há um fluxo operacional recorrente
- o conteúdo tem dono e função claros
- juntar em arquivo existente pioraria a legibilidade

## Quando editar arquivo existente
Editar arquivo existente quando:
- o novo conteúdo é continuação natural
- o arquivo já é a fonte de verdade daquele assunto
- a mudança atualiza o estado do sistema

## Quando algo vai para MEMORY
Mandar para MEMORY quando for:
- decisão relevante
- preferência importante do Alex
- contexto estável de negócio
- aprendizado recorrente
- algo que precisa sobreviver a novas sessões

## Quando algo vira skill
Algo vira skill quando:
- existe repetição clara
- a execução se beneficia de instruções especializadas
- há risco recorrente de erro
- o fluxo pode ser reutilizado por qualquer agente no sistema

## Quando algo vira doc de produto
Criar documento de produto quando existir:
- app
- sistema
- oferta estruturada
- MVP
- backlog de produto
- módulo com regras específicas

## Quando algo vira PRIORITY vs TASK
- PRIORITY = direção estratégica em andamento
- TASK = execução concreta e fila de trabalho

## Regras de limpeza
Evitar:
- duplicação de contexto
- arquivos que só repetem outro arquivo
- documentos criados por impulso sem uso operacional
- excesso de "readmes" e arquivos auxiliares desnecessários

## Regra de manutenção
Revisar periodicamente:
- arquivos sem uso claro
- arquivos duplicados
- decisões que deveriam estar em MEMORY
- tarefas que já viraram contexto estável
- ideias que precisam virar documento de produto ou ser descartadas

## Filosofia final
O segundo cérebro deve funcionar como sistema de direção e execução.

Se não ajuda a agir melhor, lembrar melhor ou decidir melhor, não entra.

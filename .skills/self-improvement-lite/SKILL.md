# Self Improvement Lite

Use this skill to record practical lessons without turning improvement into a meta rabbit hole.

## Goal

Keep useful operational learnings from being lost, while avoiding overreach, self-modification, or noisy journaling.

## When to use

Use this skill when:
- a command fails in a meaningful way
- the user corrects a wrong assumption or answer
- a better repeatable workflow is discovered
- a missing capability or integration is identified
- an external system behaves differently than expected
- a lesson should be preserved for later work

## What to capture

Capture only things that are likely to matter again:
- recurring setup gotchas
- environment quirks
- corrected assumptions
- workflow improvements
- integration blockers
- useful future enhancements

## What not to capture

Do not log:
- every minor mistake
- emotional self-talk
- vague aspirations
- internal monologue
- speculative “maybe useful” trivia
- anything sensitive that should not live in normal project notes

## Storage rule

Prefer concise notes in the appropriate operational file or daily memory file.

Good targets include:
- daily memory notes
- task-specific docs
- a relevant skill file when the lesson belongs in a reusable workflow
- project operating docs when the lesson changes how work should be done

## Decision rule

Before recording a learning, ask:
1. Will this likely matter again?
2. Is there a clear place where future me would look for it?
3. Is the lesson concrete enough to be useful?

If the answer is no, skip it.

## Output style

When surfacing a learning, keep it short:
- what happened
- what was learned
- where it should be recorded

## Hard boundary

This skill does not authorize automatic behavior changes, policy changes, or self-directed goal expansion.
It is for disciplined operational learning only.
